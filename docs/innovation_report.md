# SilverLink 项目技术创新点

## 1.3.1 技术创新点

### 创新点一：基于多层时序特征的端侧轻量级跌倒检测算法

#### 1. 技术难题分析

智能手机端跌倒检测是老年安全领域的关键技术挑战，现有解决方案主要面临以下技术瓶颈：

**（1）误报率与灵敏度的平衡困境**

传统方案采用单一加速度阈值（如冲击峰值>3G）进行跌倒判断，但日常动作如快速放下手机（2.5-4G）、坐下（2-3G）、上下楼梯（2-3G）等均可能超越阈值，导致大量误报。若提高阈值以降低误报，则会漏报老年人跌倒在沙发、床铺等软表面的情况（冲击峰值通常仅1.3-2G）。

**（2）时序模式识别缺失**

单帧加速度判断忽略了跌倒的时序特征。真实跌倒过程具有明显的阶段性特征：自由落体期（加速度接近0）→撞击期（加速度突增）→静止期（加速度稳定在重力值附近），而放下手机等动作缺乏明显的自由落体阶段和后续静止期。

**（3）端侧计算资源受限**

手机端需持续运行检测服务，对功耗和计算效率有严格要求，无法部署复杂深度学习模型。

#### 2. 技术方案设计

本项目提出**基于滑动窗口的多层时序特征分析算法**，采用"预筛选→特征提取→ML分类→静止确认"的四阶段流水线架构：

#### 3. 关键技术实现

**（1）自适应灵敏度机制**

针对不同行动能力的老年人提供三档灵敏度配置：

| 灵敏度等级 | 自由落体阈值 | 撞击阈值 | 静止阈值 | 适用场景 |
|-----------|-------------|---------|---------|---------|
| 高 | 3.5 m/s² | 25.0 m/s² | 2.0 m/s² | 行动能力差，易跌倒 |
| 中（默认） | 1.5 m/s² | 38.0 m/s² | 1.5 m/s² | 一般老年人 |
| 低 | 1.0 m/s² | 45.0 m/s² | 1.0 m/s² | 仅检测严重摔倒 |

**（2）双路径快速响应机制**

- **ML特征路径**：适用于一般跌倒场景，通过多维特征综合判断
- **极端冲击路径**：当瞬时加速度超过40 m/s²时，直接触发确认流程，避免硬着陆漏报

**（3）资源优化设计**

- 采用预筛选机制，90%以上的正常活动在第二层即被过滤
- 特征计算使用增量更新，避免重复遍历缓冲区
- 服务采用PARTIAL_WAKE_LOCK确保后台持续运行

#### 4. 技术创新要点总结

1. **多层时序特征分析**：区别于单阈值方案，本算法通过2秒滑动窗口提取13维统计与时序特征，识别跌倒的完整时序模式
2. **双路径快速响应**：同时支持ML分类和极端冲击快速通道，兼顾软着陆和硬着陆检测
3. **四阶段流水线架构**：预筛选→特征提取→ML分类→静止确认，在降低误报率的同时保持高灵敏度
4. **自适应灵敏度配置**：根据老年人行动能力动态调整检测参数

### 创新点二：基于VAD的全双工语音交互软件回声消除策略

#### 1. 技术难题分析

老年陪伴AI需要实现"随时可打断"的自然语音交互体验，但在不使用专业回声消除硬件（如双工通话设备）的普通智能手机上，面临以下技术挑战：

**（1）回声干扰问题**

当AI语音通过手机扬声器播放时，麦克风会同时采集到播放内容。若直接进行语音活动检测（VAD），会将AI的回声误判为用户语音，导致AI自我打断或产生错误识别。

**（2）打断响应延迟**

传统方案采用"半双工"模式，即AI播放期间禁用麦克风输入，用户需等待AI说完才能发言。这种模式不符合自然对话习惯，交互体验差。

**（3）误触发过滤难度**

环境噪音、电视声音、其他人说话等都可能触发VAD，如何在允许打断的同时避免误触发是核心难点。

#### 2. 技术方案设计

本项目将硬件级回声消除能力与端侧实时对话耦合，通过共享会话解决该问题，并在此之外额外设计**多重屏障的软件回声消除策略**，采用"状态感知+动态阈值+连续帧确认"的三层过滤机制：

```
┌─────────────────────────────────────────────────────────────────────────┐
│                      全双工语音交互回声消除架构                          │
├─────────────────────────────────────────────────────────────────────────┤
│  状态机 (ConversationState):                                            │
│  ┌───────────────────────────────────────────────────────────────────┐  │
│  │                                                                   │  │
│  │    ┌──────────┐    用户说话    ┌────────────┐                    │  │
│  │    │  Idle    │ ─────────────→ │ Listening  │                    │  │
│  │    └──────────┘                └────────────┘                    │  │
│  │         ↑                            │                            │  │
│  │         │                     静音超时450ms                       │  │
│  │    播放完成                          ↓                            │  │
│  │         │                    ┌────────────┐                      │  │
│  │    ┌──────────┐              │ Processing │                      │  │

│  │         ↓                                                         │  │

│  │   └─────────────┘                                                │  │

├─────────────────────────────────────────────────────────────────────────┤
│  第一层：播放冷却期过滤                                                 │
│  ┌───────────────────────────────────────────────────────────────────┐  │
│  │ 播放开始后500ms内:                                                │  │
│  │   → 完全忽略所有语音检测结果                                      │  │
│  │   → 原因：播放刚开始时回声最强，此时检测100%是误报                │  │

├─────────────────────────────────────────────────────────────────────────┤
│  第二层：动态阈值过滤                                                   │
│  ┌───────────────────────────────────────────────────────────────────┐  │
│  │ 正常监听模式 (isSpeaking = false):                                │  │
│  │   → 仅需 WebRTC VAD 检测为语音即可                                │  │

│  │ AI播放模式 (isSpeaking = true, 冷却期后):                         │  │
│  │   → 必须同时满足:                                                 │  │

│  │     } else vadSpeech                                              │  │
│  │                                                                   │  │
│  │     if (isSpeaking && isSpeech) {                                 │  │
│  │         consecutiveSpeechFrames++                                 │  │
│  │         if (consecutiveSpeechFrames < 3) return                   │  │
│  │     }                                                             │  │
│  │     ...                                                           │  │
```

#### 3. 关键技术实现

**（1）WebRTC VAD引擎集成**

采用WebRTC VAD作为基础语音活动检测器：
- 支持10ms/20ms/30ms帧长，本项目使用20ms（320采样点@16kHz）
- 提供多级灵敏度配置
- 计算开销极低，适合实时处理

**（2）RMS能量阈值计算**

```kotlin
private fun calculateRms(frame: ShortArray): Int {
    if (frame.isEmpty()) return 0
    var sum = 0.0
    frame.forEach { sample ->
        val value = sample.toInt()
        sum += value * value
    }
    return kotlin.math.sqrt(sum / frame.size).toInt()
}
```

通过实验确定阈值4000，该值显著高于典型手机扬声器回声的RMS（约1500-2500），而正常人声距离麦克风20-50cm时RMS通常在5000以上。

**（3）状态感知阈值切换**

```kotlin
val isSpeech = if (isSpeaking.get()) {
    val inCooldown = System.currentTimeMillis() - playbackStartTimeMs < playbackCooldownMs
    if (inCooldown) false  // 冷却期内完全忽略
    else vadSpeech && rms >= bargeInRmsThreshold  // 双重条件
} else {
    vadSpeech  // 正常监听，仅需VAD
}
```

#### 4. 技术创新要点总结

1. **纯软件回声消除**：无需硬件AEC支持，通过状态机+动态阈值+连续帧确认实现回声过滤
2. **三层过滤架构**：冷却期过滤（500ms）→ 动态阈值过滤（RMS≥4000）→ 连续帧确认（3帧），层层递进降低误触发
3. **状态感知阈值切换**：根据AI是否正在播放动态调整检测严格程度，平衡打断灵敏度与误触发率

### 创新点三： 声音复刻的端云直传链路
**技术要点**：将声音复刻训练音频上传改为端侧直传云存储，通过云端签名授权与直链 URL/代理下载并行方案，规避云函数大小限制并提升第三方接口兼容性。

**实现细节**：
- **直传 COS**：由云函数下发上传凭证（PUT 签名），客户端直接上传音频到 COS，避免云函数 6MB 上传限制。
- **双 URL 策略**：优先使用不含签名参数的 COS 直链 URL（便于第三方模型训练接口识别），当直链不可用时自动回退代理下载 URL。
- **端云分工清晰**：云端负责授权与安全管控，客户端负责大文件上传与回调处理，降低整体失败率。

**创新性说明**：
以“端侧直传 + 直链/代理双通道”解决语音复刻训练素材上传的限制与兼容问题，属于工程架构层面的有效创新，直接提升核心能力可用性。

### 创新点四：基于帧间差分的老年友好自动拍摄稳定检测

#### 1. 技术难题分析

在老年健康管理场景中，药品识别、体检报告拍摄等功能需要老年人使用手机相机采集图像。但老年用户面临以下挑战：

**（1）手部震颤导致图像模糊**

老年人（尤其是帕金森症等患者）手部可能存在震颤，导致拍摄时画面持续抖动，传统手动点击拍照难以捕获清晰图像。

**（2）操作界面复杂**

标准相机应用需要用户找到拍照按钮并精准点击，对视力下降、反应迟缓的老年人造成操作负担。

**（3）拍摄时机判断困难**

老年人难以判断何时画面足够稳定可以拍照，经常在手抖时按下快门导致模糊。

#### 2. 技术方案设计

本项目设计**基于像素差分的自动触发拍摄机制**：当检测到画面持续稳定超过阈值时间时，自动完成图像捕获，无需用户点击。

```
┌─────────────────────────────────────────────────────────────────────────┐
│                      自动稳定拍摄算法架构                                │
├─────────────────────────────────────────────────────────────────────────┤
│  相机预览帧流                                                           │
│       ↓                                                                 │
│  ┌───────────────────────────────────────────────────────────────────┐  │
│  │ 第一步：降采样压缩                                                 │  │
│  │   • 输入: 相机原始帧 (如 1920x1080)                               │  │
│  │   • 输出: 8x8 缩略图 (64像素)                                     │  │
│  │   • 目的: 降低计算量，同时保留整体运动信息                        │  │
│  └───────────────────────────────────────────────────────────────────┘  │
│       ↓                                                                 │
│  ┌───────────────────────────────────────────────────────────────────┐  │
│  │ 第二步：帧间差分计算                                               │  │
│  │                                                                   │  │
│  │   for (i in 0 until 64) {                                         │  │
│  │       val rDiff = |R1[i] - R2[i]|                                 │  │
│  │       val gDiff = |G1[i] - G2[i]|                                 │  │
│  │       val bDiff = |B1[i] - B2[i]|                                 │  │
│  │       if (rDiff > 30 || gDiff > 30 || bDiff > 30) {               │  │
│  │           changedPixels++                                         │  │
│  │       }                                                           │  │
│  │   }                                                               │  │
│  │   motionScore = changedPixels / 64.0  // 变化像素比例             │  │
│  └───────────────────────────────────────────────────────────────────┘  │
│       ↓                                                                 │
│  ┌───────────────────────────────────────────────────────────────────┐  │
│  │ 第三步：稳定状态判断                                               │  │
│  │                                                                   │  │
│  │   isStable = (motionScore < 0.02)  // 变化<2%视为稳定             │  │
│  │                                                                   │  │
│  │   if (isStable) {                                                 │  │
│  │       if (!wasStable) stableStartTime = now  // 记录稳定起始时间  │  │
│  │       else if (now - stableStartTime >= 500ms) {                  │  │
│  │           return TRIGGER_CAPTURE  // 稳定500ms，触发拍摄          │  │
│  │       }                                                           │  │
│  │   } else {                                                        │  │
│  │       stableStartTime = 0  // 重置计时                            │  │
│  │   }                                                               │  │
│  └───────────────────────────────────────────────────────────────────┘  │
│       ↓                                                                 │
│  ┌───────────────────────────────────────────────────────────────────┐  │
│  │ 用户反馈: 稳定进度条                                               │  │
│  │                                                                   │  │
│  │   getStabilityProgress() = elapsed / 500ms                        │  │
│  │                                                                   │  │
│  │   UI显示: ████████░░░░░░ 60%                                      │  │
│  │   提示: "请保持稳定..."                                           │  │
│  └───────────────────────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────────────────────┘
```

#### 3. 关键技术参数

| 参数 | 值 | 说明 |
|-----|-----|-----|
| SAMPLE_SIZE | 8x8 | 采样尺寸，平衡计算效率与检测精度 |
| motionThreshold | 0.02 (2%) | 像素变化比例阈值，低于此值视为稳定 |
| stableDurationMs | 500ms | 持续稳定时间阈值 |
| channelThreshold | 30 | 单通道差异阈值（0-255） |

#### 4. 技术创新要点总结

1. **免触控自动拍摄**：老年人只需将手机对准目标，系统自动在画面稳定时完成拍摄
2. **实时稳定度反馈**：通过进度条可视化稳定进度，引导用户保持手部稳定
3. **轻量级算法设计**：8x8降采样+简单像素差分，每帧计算量极低，不影响相机预览流畅度

### 创新点五：基于大模型语义理解的认知评估答案验证机制

#### 1. 技术难题分析

记忆力认知评估需要验证老年人对照片相关问题的回答正确性，但传统字符串匹配方案面临以下问题：

**（1）同义词表达**

老年人可能使用不同词汇描述相同事物：
- 预期答案："北京"
- 用户回答："首都"、"帝都"、"咱们首都"
- 字符串匹配结果：错误 ✗

**（2）口语化表达**

老年人语音输入常带有口语化特征：
- 预期答案："妈妈"
- 用户回答："俺娘"、"我妈"、"母亲大人"
- 字符串匹配结果：错误 ✗

**（3）模糊正确**

部分回答虽不完全匹配但可视为正确：
- 预期答案："2023年春节"
- 用户回答："去年过年"（假设当前为2024年）
- 应判定为：正确 ✓

#### 2. 技术方案设计

本项目设计**大模型语义匹配+关键词兜底**的双层验证架构：

```
┌─────────────────────────────────────────────────────────────────────────┐
│                      认知评估答案验证架构                                │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  用户语音回答                                                           │
│       ↓                                                                 │
│  ┌───────────────────────────────────────────────────────────────────┐  │
│  │ ASR 语音转文字                                                     │  │
│  │   Qwen-Audio 模型                                                 │  │
│  └───────────────────────────────────────────────────────────────────┘  │
│       ↓                                                                 │
│  ┌───────────────────────────────────────────────────────────────────┐  │
│  │ 第一层：大模型语义匹配 (matchAnswerWithAI)                         │  │
│  │                                                                   │  │
│  │   Prompt:                                                         │  │
│  │   """                                                             │  │
│  │   你是一个语义匹配判断器。                                        │  │
│  │   用户回答: {userAnswer}                                          │  │
│  │   参考答案: {expectedAnswers}                                     │  │
│  │   问题类型: {PEOPLE/TIME/PLACE/EVENT}                             │  │
│  │                                                                   │  │
│  │   判断规则:                                                       │  │
│  │   - 考虑同义词、口语化表达、地方方言                              │  │
│  │   - 时间问题允许相对表达（去年=2023年）                           │  │
│  │   - 人物问题考虑称谓变体（妈妈=母亲=娘）                          │  │
│  │                                                                   │  │
│  │   返回JSON: {"match": true/false, "confidence": 0.0-1.0}          │  │
│  │   """                                                             │  │
│  │                                                                   │  │
│  │   → 成功: 返回匹配结果                                            │  │
│  │   → 失败: 进入兜底层                                              │  │
│  └───────────────────────────────────────────────────────────────────┘  │
│       ↓ (网络失败/API异常时)                                            │
│  ┌───────────────────────────────────────────────────────────────────┐  │
│  │ 第二层：关键词兜底匹配 (fallbackMatch)                             │  │
│  │                                                                   │  │
│  │   fun fallbackMatch(userAnswer, expectedAnswers): Boolean {       │  │
│  │       return expectedAnswers.any { expected ->                    │  │
│  │           // 双向包含检查                                         │  │
│  │           userAnswer.contains(expected) ||                        │  │
│  │           expected.contains(userAnswer)                           │  │
│  │       }                                                           │  │
│  │   }                                                               │  │
│  │                                                                   │  │
│  │   示例:                                                           │  │
│  │   - 预期"北京", 回答"在北京拍的" → contains → 正确              │  │
│  │   - 预期"爷爷奶奶", 回答"奶奶" → contains → 正确                 │  │
│  └───────────────────────────────────────────────────────────────────┘  │
│       ↓                                                                 │
│  ┌───────────────────────────────────────────────────────────────────┐  │
│  │ 结果分类:                                                         │  │
│  │                                                                   │  │
│  │   QuizResult.Correct        → 完全正确                            │  │
│  │   QuizResult.PartiallyCorrect → 部分正确 (confidence < 0.8)       │  │
│  │   QuizResult.Incorrect      → 错误                                │  │
│  └───────────────────────────────────────────────────────────────────┘  │
│       ↓                                                                 │
│  ┌───────────────────────────────────────────────────────────────────┐  │
│  │ 认知数据记录:                                                     │  │
│  │                                                                   │  │
│  │   CognitiveLogRequest {                                           │  │
│  │       photoId, questionType,                                      │  │
│  │       expectedAnswer, actualAnswer,                               │  │
│  │       isCorrect, responseTimeMs,                                  │  │
│  │       confidence                                                  │  │
│  │   }                                                               │  │
│  │   → 上传云端 → 生成认知趋势报告                                   │  │
│  └───────────────────────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────────────────────┘
```

#### 3. 问题类型与期望答案生成

根据照片元数据动态生成不同类型的认知测试问题：

| 问题类型 | 示例问题 | 预期答案来源 |
|---------|---------|-------------|
| PEOPLE | "这张照片里有谁？" | 照片people字段 |
| TIME | "这张照片是什么时候拍的？" | 照片takenDate字段 |
| PLACE | "这张照片是在哪里拍的？" | 照片location字段 |
| EVENT | "这张照片是什么场合？" | AI描述中提取 |

#### 4. 技术创新要点总结

1. **语义级答案匹配**：利用大模型理解能力，支持同义词、口语化、方言等多种表达变体
2. **分级置信度评估**：输出0-1置信度分数，支持"部分正确"的中间状态判定
3. **双层降级机制**：大模型失败时自动降级到关键词匹配，确保功能可用性
4. **认知趋势分析**：记录答题响应时间、正确率等指标，支持长期认知能力追踪
